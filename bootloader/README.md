### Загрузчик для K1921VG015

#### 1. Назначение

Этот каталог содержит **загрузчик для микроконтроллера K1921VG015**, размещённый во внутренней Flash‑памяти и обеспечивающий обновление основной программы по интерфейсу **UART4** без использования внешнего программатора.

Загрузчик:
- принимает от хоста образ приложения (заголовок + бинарные данные);
- стирает и переписывает область Flash, отведённую под приложение;
- проверяет целостность по **CRC‑32**;
- передаёт управление приложению при успешной проверке.

---

### 2. Структура каталога `bootloader`

- `Makefile` – скрипт сборки загрузчика.
- `include/`  
  - `bl_config.h` – конфигурация адресов, размеров и кодов протокола.  
  - `bl_image.h` – описание заголовка образа и функции проверки валидности.  
  - `bl_crc32.h`, `bl_hal.h`, `bl_jump.h` и др. – интерфейсы аппаратно‑зависимых модулей.
- `src/`  
  - `bl_main.c` – точка входа, логика выбора между режимом обновления и запуском приложения, реализация протокола.  
  - `bl_hal.c` – инициализация UART4, GPIO, операции с Flash.  
  - `bl_image.c` – работа с заголовком и образом приложения.  
  - `bl_crc32.c` – вычисление CRC‑32 через аппаратный блок CRC0.  
  - `bl_jump.c` – безопасный переход в приложение.  
  - `bl_trap.c` – обработка исключений (если используется).
- `device/ldscripts/k1921vg015_boot.ld` – линкерный скрипт загрузчика.
- `build/` – артефакты сборки (`.elf`, `.hex`, `.dump`).

---

### 3. Распределение памяти

**Область загрузчика**

- `BL_BASE_ADDR = 0x80000000`  
- `BL_SIZE_BYTES = 0x00002000` (8 КБ)

**Область приложения**

- `APP_BASE_ADDR = 0x80002000`  
- `APP_MAX_SIZE_BYTES = 0x000FE000`  
- `APP_END_ADDR = APP_BASE_ADDR + APP_MAX_SIZE_BYTES`

**Формат образа приложения**

В начале области приложения размещается заголовок:

```c
typedef struct {
    uint32_t image_size;  // размер полезной части
    uint32_t crc32;       // CRC-32 по полезной части
} bl_app_header_t;
```

- `APP_HEADER_ADDR = APP_BASE_ADDR`  
- `APP_PAYLOAD_ADDR = APP_HEADER_ADDR + 8`  
- `APP_PAYLOAD_MAX_SIZE_BYTES = APP_MAX_SIZE_BYTES - 8`  
- `APP_ENTRY_ADDR = APP_PAYLOAD_ADDR` – точка входа приложения.

---

### 4. Режимы работы и логика старта

1. При старте МК вызывается `main()` загрузчика.
2. `bl_hal_init()` инициализирует:
   - `UART4` (пины `PA8` – RX, `PA9` – TX, 115200 8N1);
   - кнопку обновления на `PC0` (активный низ);
   - светодиоды режима обновления на `PA12..PA15`.
3. Если кнопка обновления нажата – сразу вход в **режим обновления**.
4. Если кнопка не нажата:
   - при валидном образе (`bl_image_is_valid()` возвращает `true`) – переход в приложение `bl_jump_to_app(APP_ENTRY_ADDR)`;
   - при невалидном образе – вход в режим обновления.

В режиме обновления светодиоды включены, загрузчик по UART4 обменивается данными с хостом до успешной записи и запуска нового образа.

---

### 5. Протокол обновления по UART

**Транспорт**

- Интерфейс: `UART4`  
- Скорость: `115200` бод  
- Формат: `8N1`  
- Таймаут приёма: `BL_UPDATE_WAIT_TIMEOUT_MS = 500` мс на заголовок и каждый блок.

**Последовательность**

1. Загрузчик периодически отправляет байт состояния:
   - `BL_PROTO_REPLY_WAITING = 1`
2. Хост отправляет **заголовок** (`bl_app_header_t`, 8 байт):
   - `image_size` – размер полезной части;
   - `crc32` – ожидаемое CRC‑32 (может быть 0 для вычисления на устройстве).
3. Загрузчик:
   - проверяет размер (`0 < image_size ≤ APP_PAYLOAD_MAX_SIZE_BYTES`), иначе отправляет `BL_PROTO_ERR_TOO_BIG = 3`;
   - стирает область приложения через `bl_hal_flash_erase_range()`. При ошибке отправляет `BL_PROTO_ERR_WAIT_ERASE_PAGE = 11`.
4. При успешном стирании отправляет `BL_PROTO_REPLY_ACK = 2`.
5. Хост передаёт полезную нагрузку (прошивку) блоками до 256 байт.
6. Для каждого блока:
   - при успехе записи – `BL_PROTO_REPLY_ACK = 2`;
   - при ошибке приёма – `BL_PROTO_ERR_RECEIVE = 4`;
   - при ошибке записи – `BL_PROTO_ERR_WAIT_WRITE_PAGE = 7`.
7. После передачи всех данных загрузчик:
   - считает CRC‑32 по полезной части (аппаратный CRC0);
   - если `crc32 == 0` в заголовке – подставляет вычисленное значение;
   - если `crc32 != 0` и не совпадает – отправляет `BL_PROTO_ERR_CRC32 = 6`.
8. При успешной проверке:
   - записывает заголовок во Flash;
   - повторно проверяет образ через `bl_image_is_valid()` и `bl_image_get_size()`;
   - несколько раз отправляет `BL_PROTO_REPLY_ACK`, ждёт опустошения TX FIFO и выполняет `bl_jump_to_app(APP_ENTRY_ADDR)`.

### 6. Сборка

**Зависимости**

- тулчейн RISC‑V (`riscv-gcc`, `objcopy`, `objdump`, `size`, `gdb`);
- переменная окружения `CROSS_PREFIX` должна указывать на префикс тулчейна (например, `riscv-none-elf-`).

**Основные цели Makefile**

В каталоге `bootloader/`:

- сборка всего:

```bash
make all
```

Результат:
- `build/bootloader.elf` – исполняемый файл;
- `build/bootloader.hex` – Intel HEX для прошивки;
- `build/bootloader.dump` – дизассемблированный дамп;
- вывод размеров секций (`make size` внутри `all`).

- только `.hex` (если ELF уже собран):

```bash
make hex
```

- просмотр размера:

```bash
make size
```

- очистка:

```bash
make clean
```

---

### 7. Требования к приложению

Чтобы приложение корректно запускалось через загрузчик, необходимо:

- линковать его с началом кода по адресу `APP_ENTRY_ADDR` (`0x80002008`), обычно через отдельный линкерный скрипт;
- формировать в начале Flash заголовок `bl_app_header_t`:
  - `image_size` – точный размер полезной части в байтах;
  - `crc32` – CRC‑32 по полезной части (или 0, если значение должен заполнить загрузчик);
- гарантировать, что общий размер (`8` байт заголовка + `image_size`) не превышает `APP_MAX_SIZE_BYTES`.

---

### 8. Аппаратные соединения

- `UART4_TX` – `PA9` → RX‑вход хоста;
- `UART4_RX` – `PA8` ← TX‑выход хоста;
- кнопка входа в режим обновления – `PC0`, активный низ (замыкание на GND);
- светодиоды индикации режима обновления – `PA12..PA15`.
